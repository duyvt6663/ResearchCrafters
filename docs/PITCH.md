# ResearchCrafters Pitch

## One-Liner

ResearchCrafters is CodeCrafters for AI research papers: learners rebuild famous papers as
expert-crafted executable research journeys with decisions, failed branches, experiments,
implementation tasks, evidence, and writing feedback.

## Problem

Papers hide the actual research process. They present a clean final narrative, but omit the
failed hypotheses, implementation traps, experimental tradeoffs, and evidence calibration
that make someone a strong research engineer.

AI engineers can read papers and copy repositories, but many still struggle to:

- choose the right technical branch,
- design useful experiments,
- interpret failure,
- connect evidence to claims,
- and write defensible research arguments.

## Solution

ResearchCrafters turns a famous paper into an Executable Research Package.

Each package guides the learner through the research journey:

- the canonical path,
- plausible failed and suboptimal paths,
- implementation stages,
- miniature experiments,
- evidence interpretation,
- claim-writing tasks,
- deterministic tests,
- and AI mentor feedback grounded in expert-authored content.

The learner does not just read what worked. They practice deciding what to try and learn
why other branches failed.

## Why Now

AI has made code generation easier, but it has made judgment more valuable. When everyone
can ask an assistant to implement an idea, the differentiator is knowing which idea is
worth implementing, how to test it, and what the results actually prove.

At the same time, agent-native research artifacts show that papers can be represented as
structured, executable knowledge rather than static PDFs. ResearchCrafters adapts that
idea for human learning.

## Product

The product has three surfaces:

1. Learner platform: catalog, stage player, decision graph, progress, feedback, and
   shareable results.
2. Local/remote execution: starter repo, CLI submission, tests, mini-experiments, and run
   logs.
3. Authoring system: ERP schema, package validator, graph editor, evidence manager, and
   expert review workflow.

## Moat

The moat is expert reconstruction of research process.

The hard content is not the paper summary. The hard content is knowing:

- which decisions mattered,
- which branches looked plausible,
- which failures are educational,
- which implementation details are critical,
- which claims are actually supported,
- and how to turn all of that into a coherent learning journey.

Agents can draft. Experts must curate.

## Wedge

Start with one flagship package for a famous AI paper and make it excellent.

Good candidates:

- FlashAttention for AI systems engineers.
- ResNet for cheaper, clear evidence-based reconstruction.
- DPO/RLHF for LLM alignment and objective reasoning.

The first package should be strong enough that users share: "I thought I understood this
paper, then this exposed the gap."

## Business Model

Individual subscription:

- Free onboarding and preview stages.
- Paid access for full packages, mentor feedback, code examples, and history.

Team plan:

- Seats, team analytics, private cohorts, and training paths.

Institutional plan:

- Course use, assignment tracking, and instructor dashboards.

## 90-Day Plan

Days 1-15:

- Finalize ERP schema and package authoring workflow.
- Pick the first flagship paper.
- Produce a static prototype of the learning session.

Days 16-45:

- Build package catalog, learner session state, first package stages, and deterministic
  validation.
- Build the CLI or simplest viable runner path.

Days 46-70:

- Add mentor feedback, progress tracking, share cards, and package validation.
- Run alpha with 20-50 serious AI engineers.

Days 71-90:

- Improve the flagship package from cohort data.
- Add payment.
- Publish public decision challenges.
- Recruit the next two package authors or reviewers.

## Sharp Version

ResearchCrafters is a research-thinking gym. CodeCrafters proved engineers will pay to
rebuild real systems in a staged, test-driven format. ResearchCrafters applies that loop to
the harder skill AI engineers now need: research judgment. Each famous paper becomes an
executable package where learners make the decisions, run the experiments, inspect the
evidence, and learn from the failures the paper left out.
